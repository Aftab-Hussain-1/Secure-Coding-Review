CRITICAL DESER-001 Insecure Deserialization (pickle) ▾ Deserialization CWE-502 vulnerable_app.py:11 MEDIUM

📋 Description

The pickle module is used to deserialize data from untrusted sources. Maliciously crafted pickle data can execute arbitrary Python code during deserialization, leading to full system compromise.

📍 Vulnerable Code (line 11)

import pickle

🔴 Vulnerable Pattern

session_data = pickle.loads(request.data)

✅ Secure Alternative

session_data = json.loads(request.get_data(as_text=True))

🛠️ Remediation Steps

Never deserialize pickle data from untrusted sources.
  BAD:  data = pickle.loads(request.data)
  GOOD: data = json.loads(request.data)  # Use JSON instead

Prefer JSON, MessagePack, or Protocol Buffers for data interchange.
If pickle is unavoidable, cryptographically sign and verify data
before deserializing using hmac.

CRITICAL INJ-001 SQL Injection ▾ Injection CWE-89 vulnerable_app.py:31 HIGH

📋 Description

SQL queries are constructed using string concatenation or f-strings with user-controlled input, allowing attackers to manipulate the query structure and access or modify unauthorized data.

📍 Vulnerable Code (line 31)

query = f"SELECT * FROM users WHERE username='{username}' AND password='{password}'"

🔴 Vulnerable Pattern

✅ Secure Alternative

cursor.execute("SELECT * FROM users WHERE name=?", (username,))

🛠️ Remediation Steps

Use parameterized queries (prepared statements) exclusively.
  BAD:  cursor.execute(f"SELECT * FROM users WHERE id='{user_id}'")
  GOOD: cursor.execute('SELECT * FROM users WHERE id=?', (user_id,))

For ORMs (SQLAlchemy), use query filters rather than raw SQL strings.
Apply input validation and allowlisting where possible.
Use stored procedures with proper parameter handling.
Apply the principle of least privilege to database accounts.
